# OCI OKE Detailed Prerequisites Guide

This guide expands the prerequisite checklist used by the OKE deployment runbook and focuses on first-run operator setup.

Scope:
- OCI account and access readiness
- OCI CLI install and configuration
- OKE cluster access with kubectl
- Docker and Docker buildx readiness
- OCIR auth token and registry login

## 1. OCI Account, User, and Compartment Readiness

Before running deploy scripts, confirm these basics in the OCI Console:

1. You have an OCI tenancy and can sign in to your target region.
2. You have a target compartment where OKE and related resources exist.
3. Your user can manage the required resources in that compartment.
4. You can view an existing OKE cluster in Active state.

Recommended first-run approach:
- Use a user with broad permissions for initial setup.
- After successful setup, reduce permissions to least privilege.

Quick validation commands:

```powershell
oci iam region list
oci iam region-subscription list
```

If these fail with authorization errors, fix IAM access before proceeding.

## 2. Install and Configure OCI CLI

You have two options for running OCI CLI commands:

### Option A: Persistent OCI CLI Docker Container (recommended)

This project includes a pre-built persistent container based on Oracle's official `ghcr.io/oracle/oci-cli` image. It avoids installing OCI CLI on your host and keeps your environment clean.

Location: `deploy/docker/oci-cli/`

Start the container:

```powershell
pwsh deploy/docker/oci-cli/oci-cli.ps1 start
```

First-time OCI CLI setup (interactive):

```powershell
pwsh deploy/docker/oci-cli/oci-cli.ps1 setup
```

This runs `oci setup config` inside the container. Provide:
- User OCID
- Tenancy OCID
- Region identifier (example: us-ashburn-1)
- API key material (generated by the setup flow, or manually supplied)

The config is saved to your host `~/.oci/` directory (mounted into the container), so it persists across restarts.

Run OCI commands:

```powershell
pwsh deploy/docker/oci-cli/oci-cli.ps1 run oci --version
pwsh deploy/docker/oci-cli/oci-cli.ps1 run oci os ns get
pwsh deploy/docker/oci-cli/oci-cli.ps1 run oci iam region list
```

Open an interactive shell:

```powershell
pwsh deploy/docker/oci-cli/oci-cli.ps1 shell
```

Other actions:

```powershell
pwsh deploy/docker/oci-cli/oci-cli.ps1 status   # check container state
pwsh deploy/docker/oci-cli/oci-cli.ps1 stop     # stop (preserves state)
pwsh deploy/docker/oci-cli/oci-cli.ps1 start    # resume
```

The container uses `restart: unless-stopped`, so it survives Docker Desktop restarts and is always ready.

Volumes mounted from host:
- `~/.oci` — OCI CLI config and API keys
- `~/.kube` — kubeconfig for kubectl access

### Option B: Native OCI CLI install

Install options documented by Oracle:
- Standard installer and install verification
- Manual install from package sources

After install, configure the CLI profile:

```powershell
oci setup config
```

During setup, provide:
- User OCID
- Tenancy OCID
- Region identifier (example: us-ashburn-1)
- API key material (generated by the setup flow, or manually supplied)

### Validate CLI (either option)

```powershell
oci --version
oci os ns get
```

Expected outcome:
- oci --version returns a version string
- oci os ns get returns your tenancy namespace (used later for OCIR login and image names)

## 3. OKE Cluster Exists and Is Reachable

In OCI Console:
1. Open Kubernetes Engine.
2. Open your cluster.
3. Confirm lifecycle state is Active.
4. Copy the Cluster OCID and note region.

Generate kubeconfig (Oracle-recommended token version):

```powershell
oci ce cluster create-kubeconfig --cluster-id <cluster-ocid> --file $HOME/.kube/config --region <region-identifier> --token-version 2.0.0 --kube-endpoint PUBLIC_ENDPOINT
```

Then verify access:

```powershell
kubectl config current-context
kubectl get nodes
```

Notes:
- kubeconfig token version 2.0.0 is required for current OKE behavior.
- kubeconfig entries generated by OCI CLI are user-scoped and not intended for sharing.

## 4. kubectl Installed and Available

Verify kubectl is installed and on PATH:

```powershell
kubectl version --client
```

If kubectl is missing, install it first, then rerun kubeconfig creation.

## 5. Docker and Buildx Readiness

Your scripts now run preflight checks before builds. Confirm readiness manually:

```powershell
docker version
docker buildx version
docker buildx inspect --bootstrap
```

If no active builder exists, run:

```powershell
docker buildx create --name seriescatalog-builder --use
docker buildx inspect --bootstrap
```

Expected outcome:
- buildx inspect --bootstrap completes without errors
- builder is active for subsequent builds

## 6. OCIR Auth Token and Registry Login

Before pushing images, generate an OCI auth token:
1. Open user profile in OCI Console.
2. Open Auth Tokens.
3. Select Generate Token.
4. Copy token immediately and store it securely.

Login format:
- Registry host: <region-key>.ocir.io
- Username: <tenancy-namespace>/<username>
- Federated username format: <tenancy-namespace>/<domain-name>/<username>

Login command:

```powershell
docker login <region-key>.ocir.io
```

When prompted:
- Username: tenancy namespace plus username format above
- Password: auth token

Optional non-interactive pattern used by scripts:

```powershell
$OciAuthToken | docker login "<region-key>.ocir.io" -u "<tenancy-namespace>/<username>" --password-stdin
```

## 7. Full Preflight Check (Copy/Paste)

Run this before first deploy:

```powershell
oci --version
oci os ns get
oci ce cluster create-kubeconfig --cluster-id <cluster-ocid> --file $HOME/.kube/config --region <region-identifier> --token-version 2.0.0 --kube-endpoint PUBLIC_ENDPOINT
kubectl get nodes
docker version
docker buildx version
docker buildx inspect --bootstrap
docker login <region-key>.ocir.io
```

If all commands pass, prerequisites are complete.
